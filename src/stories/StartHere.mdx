# The starter, in one page

A reusable foundation for building content-driven websites with **Next.js 16**. Pages are written
as data, the UI is a small set of components you compose, and one set of tokens drives colour,
type and spacing. This Storybook is where you read how it works and see every piece in isolation.

_Built by Jeffrey Ullers._

## What it's for

You start a new site. Instead of wiring up routing, metadata, SEO, a styling system and a component
library from scratch each time, you get them here, already fitted together and documented:

- **Content as data** — a page is a plain object, not hand-written markup, so it's ready to move to
  a CMS later without touching the components.
- **A validated contract** — every page and block is checked against a schema, at build time *and*
  runtime, so broken content (a missing field, a rating out of range) fails instead of shipping.
- **One styling system** — colour, typography and spacing come from tokens; change them in one place
  and the whole site follows.
- **SEO and sharing out of the box** — metadata, JSON-LD, Open Graph images, a sitemap and robots
  are generated from the same content.

## The core idea

A page is **data, not markup**. You write an object — `{ meta, blocks }` — and the framework turns
it into HTML through one fixed pipeline:

```
content object        →  lib/content (validate)  →  Blocks registry   →  page (Server Component)  →  HTML
{ meta, blocks }         getPageByPath()             type → component     renders <Blocks blocks={…} />
```

Each block is `{ type, ...props }`. The `type` string selects which component renders it. Because the
data and the components are separate, the same block type can appear many times with different data,
and you reorder a page by reordering an array. **[1. Architecture](?path=/docs/for-developers-1-architecture--docs)** walks through the whole pipeline.

## How the UI is layered

A page is built from one kind of thing: **content blocks**. Each layer below is composed from the one
under it.

```
page
└─ content blocks   — the only thing that goes on a page  (Hero, FeatureCards, TextMedia, Prose, Reviews)
   └─ components     — reusable, often interactive pieces  (Card, Carousel, Modal, Tabs, Form, EventCard, …)
      └─ basics      — the smallest primitives  (Button, Title, Content, Media, Section, Icon, Pill, Interactive)
```

Separately, **`structures`** holds the page shells that wrap a page in the layout rather than sitting
inside it — the public `Navigation`, `Footer` and `AnnouncementBar` (composed by `SiteChrome`) and the
admin `MegaMenu` and `BottomTabBar`. Each layer composes only the layers beneath it: a content block
never hand-rolls a button or a card, it uses the system component — and every capability lives in the
lowest layer that can own it (the rules are spelled out in
[Conventions](?path=/docs/for-developers-2-conventions--docs)).

## The repo map

| Folder | What it holds |
|---|---|
| `src/app` | Routes (App Router): pages, the root layout, `error`/`loading`/`not-found`, and the generated files — `sitemap`, `robots`, OG images, `manifest`. |
| `src/components/<layer>` | The UI, grouped into `basics`, `components`, `contentBlocks` and the `structures` shell (see above). |
| `src/content` | The content itself: page data files in `pages/`, registered in `index.ts`. |
| `src/lib` | Non-UI logic: `content/` (async accessors + the Zod contract in `content/schema/`), `seo.ts`, `site.ts`, `env.ts` (the validated environment contract), `ogImage.tsx`. |
| `src/hooks` | Small client-side utilities (`useHaptics`, `useHotkey`, `useOverlay`). |
| `src/styles` | SCSS: `_initial.scss` (values + engine), `base.scss` (the global cascade), and `components/<layer>/` — a per-component stylesheet mirroring the component tree, all imported once through `src/styles/index.ts`. |
